에이전트 스킬(agent skills)은 현재 여러 AI 에이전트 제품이 채택하는 강력한 오픈 표준으로 발전했다.
과거 클로드 코드(claude code)는 에이전트의 행동을 확장하기 위해 **사용자 지정 명령어(Custom Commands)**라는 방식을 사용했다. `.claude/commands/` 폴더 내에 단일 마크다운(.md) 파일 형태로 프롬프트 템플릿을 저장하는 단순한 형태였다. 이는 다음과 같은 한계점이 있었다.

- 사용자가 항상 슬래시(/) 명령어를 통해 수동으로만 호출해야 했으며, AI가 스스로 판단하여 자동 실행하는 기능(Auto-triggering)이 없었다.
- 명령어가 실행될 때마다 전체 프롬프트가 한 번에 로드되어 AI의 컨텍스트 윈도우(토큰)를 크게 낭비했다.
- 스크립트를 번들링하거나 복잡한 참조 문서를 분리해서 관리할 수 없다는 구조적 한계가 있었다.

사용자 지정 명령어의 한계를 극복하기 위해 앤스로픽(anthropic)은 새로운 방식인 에이전트 스킬을 개발했고, 오픈 표준으로 공개했다. 단일 파일 방식에서 벗어나, `SKILL.md` 파일을 중심으로 한 디렉토리 구조를 새롭게 채택했다. 이를 통해 실행 스크립트(scripts/)와 참조 문서(references/)를 분리하여 체계적인 관리가 가능해졌다. 사용자 의도를 파악해 0.1초 만에 알아서 스킬을 활성화하는 자동 호출(Model-invoked) 기능이 도입되었다.

필요한 순간에만 문서를 읽어들이는 **점진적 정보 노출(Progressive Disclosure) 메커니즘**을 적용하여 컨텍스트 토큰 소비를 극적으로 줄였다. 점진적인 정보 노출 메커니즘은 에이전트가 한정된 컨텍스트 윈도우(토큰)를 극도로 효율적으로 관리하기 위해 채택한 핵심 설계 원칙이다. 이 메커니즘은 수십 개의 스킬이 설치되어 있더라도 에이전트가 모든 내용을 처음부터 다 읽지 않게 막아주며, **발견(Discovery), 활성화(Activation), 실행(Execution)**의 3단계로 정보를 나누어 필요할 때만 로드한다.

- 발견(카탈로그 로드)
	- 세션이 시작될 때, 에이전트는 사용 가능한 모든 스킬의 이름과 간단한 설명이 담긴 카탈로그를 로드한다. 이 단계에서는 스킬의 상세 내용이나 실행 방법은 포함되지 않는다.
	- 아 단계에서 스킬당 약 10~20 토큰 정도의 최소한의 정보만 소비한다. 에이전트는 어떤 스킬이 존재하고 언제 사용할 수 있는지에 대한 개요만 갖게 된다.
- 활성화(지시사항 로드)
	- 사용자의 요청이 특정 스킬의 설명과 일치하여 에이전트가 해당 스킬이 필요하다고 판단하면, 에이전트는 그 스킬의 지시사항(instructions)을 로드한다. 지시사항은 스킬을 어떻게 활성화하고 사용할 수 있는지에 대한 구체적인 정보가 담긴 부분이다.
- 실행 및 자원 활용(리소스 온디멘드 로드)
	- 스킬이 활성화된 후, 에이전트가 스킬을 실행하기 위해 필요한 리소스(예: 외부 API 호출, 참조 문서 등)를 온디멘드 방식으로 로드한다. 
	- 에이전트는 이 추가 리소스들을 한 번에 로드하지 않고, 지시사항을 수행하는 과정에서 특정 파일의 정보나 스크립트 실행 결과가 정말로 필요한 순간에만 해당 리소스를 불러온다. 이를 통해 불필요한 토큰 소비를 방지하고, 에이전트가 더 많은 스킬을 효율적으로 관리할 수 있게 된다.

## 에이전트 스킬의 구조

SKILL.md 파일은 크게 설정값을 정의하는 **YAML 프론트매터(Frontmatter)**와 에이전트가 수행할 실제 지시사항을 담은 **마크다운 본문(Markdown Body)**으로 구성된다. YAML 프론트매터에는 다음과 같은 필드들이 포함된다.

- name
    - 스킬의 고유 이름이다. 최대 64자, 소문자/숫자/하이픈만 허용한다.
    - 사용자가 /스킬이름 형태로 명시적 호출을 할 때 사용된다.
- description (required)
    - AI 에이전트가 의도를 파악해 스킬을 **자동 호출(Auto-trigger)**할 때 사용하는 핵심 필드다.
    - 50~1024자 사이로 “무엇을 하는지”, “언제 사용하는지”, “어떤 키워드에 반응할지”를 구체적으로 명시해야 한다.
- disable-model-invocation
    - true 값으로 설정하면 AI 에이전트의 자동 호출을 막고, 오직 사용자가 `/ 명령어`를 입력했을 때만 실행된다.
    - 커밋, 배포 등 원치 않게 실행되면 안 되는 작업에 유용하다.
- user-invocable
    - false 값으로 설정하면 사용자의 `/ 명령어` 메뉴에서 숨긴다.
    - 배경 지식이나 팀 규칙처럼 AI 에이전트만 백그라운드에서 참고해야 하는 스킬에 사용한다.
- allowed-tools
    - 스킬 실행 중 AI 에이전트가 권한 허용을 묻지 않고 바로 사용할 수 있는 도구(예: Bash, Read)를 화이트리스트 형태로 제한한다.
- context 및 agent
    - `context: fork` 값으로 설정하면 메인 대화창의 컨텍스트를 오염시키지 않고 완전히 격리된 새로운 컨텍스트에서 서브(sub)에이전트가 작업을 백그라운드로 실행한다.
    - 이때 agent 필드로 특정 에이전트(예: Explore)를 지정할 수 있다.

```
# SKILL.md 예시

---
name: "github-issue-creator"
description: "이 스킬은 사용자가 GitHub 이슈를 생성하도록 도와줍니다. 사용자가 'GitHub 이슈 생성'과 같은 키워드를 포함한 요청을 하면 자동으로 활성화됩니다."
disable-model-invocation: false
user-invocable: true
allowed-tools: ["Bash", "Read"]
context: "fork"
```

마크다운 본문에는 스킬이 수행할 지시사항(instructions)과 필요한 리소스(resources)가 포함된다. 지시사항은 AI 에이전트가 스킬을 어떻게 활성화하고 사용할 수 있는지에 대한 구체적인 단계별 안내를 제공한다. 리소스는 스킬 실행에 필요한 외부 API 호출, 참조 문서 등의 정보를 담는다. 

**1. 명확한 목표

```
# skill-name
Your goal is to [specific task] for [specific target].
```

**2. "언제 사용해야 하는지"에 대한 안내를 추가한다** (에이전트 및 사용자 모두에게)

```
## When to Use This Skill
Use this skill when:- A user asks to [trigger phrase 1]- You need to [trigger phrase 2]- Keywords: [keyword1], [keyword2], [keyword3]
```

**3. 요구사항을 명확하게 정의**

```
## Requirements
- Must follow [standard/pattern]- Should include [specific element]- Avoid [anti-pattern]
```

```

## Specifications
1. Step 1: [Detailed instruction for step 1]
2. Step 2: [Detailed instruction for step 2]
```

**4. 참조 번들 자산**

```
## References
- Follow patterns in [references/patterns.md](references/patterns.md)- Use template from [templates/starter.json](templates/starter.json)
```

**5. 예를 들어 설명** 

```
### Good Example[Show desired output]
### What to Avoid[Show problematic patterns]
```

구체적으로 아래와 같은 내용을 참고해야 한다.

- 에이전트가 이미 아는 일반적인 지식(HTTP 개념, GitHub 이슈 구조 등)은 과감히 생략한다. 프로젝트만의 특정 규약, 엣지 케이스, 고유한 API패턴 등 에이전트가 스스로 알 수 없는 내용만 담아 컨텍스트 토큰을 절약한다.
- 프로젝트의 특유의 주의사항을 명시한다. 예를 들어, GitHub 이슈 생성 스킬이라면, “이슈 제목은 50자 이내로 작성해야 한다”와 같은 실무적인 규칙을 명시한다.
- 복잡한 작업은 체크박스 형태로 단계별로 나누어 설명한다. 예를 들어, “1. GitHub API에서 레포지토리 정보 가져오기, 2. 이슈 제목과 본문 생성하기, 3. API 호출로 이슈 생성하기”와 같이 구체적으로 안내한다.
- 작업을 한 번에 끝내지 말고 `계획 - 검증 - 실행` 사이클을 권장한다. 예를 들어, “먼저 API 호출로 이슈를 생성한 후, 생성된 이슈 번호를 반환하면 다음 단계로 넘어가서 추가 작업을 수행한다”와 같이 단계별로 안내한다.
- 본문은 가급적 500줄 이하로 유지하며, 방대한 API명세서나 코드는 본문이 아닌 참조 문서로 분리해서 관리한다. 예를 들어, “GitHub API 명세는 `references/github-api.md` 파일을 참고한다”와 같이 안내한다.

## 스킬 폴더의 구조

- SKILL.md (중심 진입점)
    - YAML 프론트매터와 마크다운 본문으로 구성된 필수 파일이다.
    - 컨텍스트 토큰 절약을 위해 본문 길이는 가급적 500줄 이하로 유지하는 것이 권장된다.
- scripts/ (스크립트)
    - 에이전트가 직접 실행할 수 있는 Python, Bash, JavaScript 등의 코드를 분리하여 저장한다.
    - 스크립트를 사용하면 에이전트가 매번 코드를 생성하지 않아도 되므로 일관되고 결정론적인 결과를 얻을 수 있다.
- references/ (참조 문서)
    - API 명세서, 복잡한 데이터 포맷 등 부피가 큰 문서를 저장한다.
    - 에이전트는 사용자의 요청에 따라 정말로 해당 지식이 필요한 순간에만 이 파일들을 로드한다.
- assets/ (정적 리소스)
    - 문서 템플릿, 다이어그램 예시, 조회용 데이터 테이블(스키마) 등 스킬 실행이나 결과물 생성에 필요한 정적 파일들을 모아둔다.
- evals/ (평가 및 테스트)
    - 스킬이 제대로 동작하는지 검증하기 위한 테스트 프롬프트(evals.json)를 저장하여 테스트 주도 개발 방식으로 스킬의 품질을 평가한다.
